# Design Phase v1 - EDA PoC Phase 2 실행 설계

## 한눈에 결론
- Phase 설계 핵심 결론: 이번 스프린트는 `Phase-1(기반 구축 + 기본 이벤트 흐름)`만 수행하고, 정량 검증은 다음 Phase로 넘긴다.
- 확정된 결정 2~3개:
  - 이번 스프린트 포함 US는 `US-1, US-2` 두 개로 제한한다.
  - 성능 비교/장애 격리 정량 검증은 `Phase-2`에서 수행한다.
  - Sprint 상태 문서는 `/monitor-sprint`에서 초기화/갱신하고, 구현 Step 상세는 `/design-implementation`에서 작성한다.
- 이번 스프린트 집중 대상: `Phase-1 / US-1, US-2`
- 미확정/리스크: 후속 서비스 지연값(500ms~2s), k6 초기 파라미터, 로컬 자원 제한에 따른 기동 변동성

## 1) 2W 요약
- 무엇(What): Kafka 기반 EDA Fan-out PoC의 기반(프로젝트 구조, 인프라, 기본 이벤트 흐름)이 아직 없어 핵심 검증을 시작할 수 없는 문제
- 왜(Why): 2주 제약 안에서 Phase 3 정량 검증까지 이어가려면 지금 기반 구축을 확정해 실행 리드타임을 줄여야 하기 때문
- 제약 조건: 전체 PoC 2주, 1인 개발, PoC 완성도 우선
- 성공 기준: `/design-phase`에서 Phase 2 MVP 범위/US를 즉시 설계 가능한 수준으로 2W가 확정되어 GO 결정
- 경계: 이번 단계에서 구현 Step 상세를 만들지 않고, Phase/US/지표/완료조건까지만 확정

## 2) Phase 설계
| Phase | 목표 | 완료 기준 | 포함 US |
| --- | --- | --- | --- |
| Phase-1 (현재) | Kafka 기반 실행 환경과 동기/비동기 기본 흐름을 준비해 다음 검증 단계의 시작 조건을 만든다. | 멀티 프로젝트 구조/인프라 구성 기준이 확정되고, 동기/비동기 주문 요청이 기본 시나리오에서 정상 동작한다. | US-1, US-2 |
| Phase-2 (다음) | EDA 핵심 가치(응답성, 장애 격리, 메시지 유실 방지)를 정량/정성으로 검증한다. | 동기 vs 비동기 비교 데이터, 장애 격리 재현 결과, 메시지 유실율 검증 결과가 확보된다. | US-3 |

## 3) 범위 정의
| 구분 | 항목 |

## 5) 지표와 스프린트 전달 정보
### 5-1. 지표
- 선행 지표(Leading):
  - 측정 방법: 이번 스프린트 포함 US(`US-1`, `US-2`) 완료율(완료 US 수 / 2)
  - 목표: 스프린트 종료 시 100%(2/2)
- 결과 지표(Outcome):
  - 측정 방법: 동기/비동기 기본 스모크 시나리오 성공률(경로별 성공 횟수 / 시도 횟수)
  - 목표: 동기/비동기 모두 100% 성공(경로별 5/5 기준)

### 5-2. 스프린트 전달 정보
- 스프린트 목표(Sprint Goal): Phase-1 US 2개를 완료해 Phase-2 정량 검증을 바로 시작할 수 있는 실행 베이스라인을 확보한다.
- 완료 기준(Definition of Done):
  - US-1/US-2 완료 기준이 모두 충족된다.
  - 범위 In 항목 3개가 모두 충족되고 Out 항목이 유입되지 않는다.
  - Leading/Outcome 지표가 목표값을 충족하거나, 미달 시 원인/재계획이 기록된다.
- 선행 의존사항:
  - Docker/Compose 실행 가능 환경
  - JDK/Gradle 실행 가능 환경
  - Kafka/서비스 포트 충돌 없음
  - `loop-v0` 학습 산출물(개념/패턴) 참조 가능 상태
- 상태 문서 작성/갱신 위치: `/monitor-sprint` (`02-sprint-status.md`)
- 구현 Step 작성 위치: `/design-implementation` 단계
