## questions.md 요약

- **What**: EDA PoC - 이벤트 기반 비동기 처리, Kafka 메시징, 시스템 간 느슨한 결합
- **Why**: 이직용 + 학습용 (JD 키워드: "MSA 경험 우대", "Kafka 사용 경험")

## 첫 접근 (How부터 시작한 실수)

### 초기: "EDA + MSA를 동시에 하나의 프로젝트로"

**사용자 초기 생각:**
- EDA + MSA를 동시에 진행
- Spring MSA 라이브러리(Gateway, Eureka 등) 쇼케이스
- 주문 + 재고를 EDA 및 Kafka, MSA로 분리

**문제점:**
- ❌ EDA, MSA 아키텍처, Spring Cloud 인프라 = **3개의 별개 관심사**가 섞여 있음
- ❌ 2주에 3가지 모두 → **~17일 예상 (2주 초과)**
- ❌ "Spring MSA 라이브러리 쇼케이스"가 모호함 (Gateway? Eureka? Config? Circuit Breaker?)
- ❌ 이전 동시성 PoC에서 "3가지 모두 → 각각 얕아짐" 교훈 반복 위험

### 가설 검증: "EDA 없는 MSA는 빈 껍데기"

**결론:** 가설이 아닌 재정의 필요
- 많은 MSA는 REST API 동기 통신 기반 (EDA 없이도 MSA 가능)
- 정확한 표현: "서비스 간 통신 방식을 보여줘야 MSA가 의미 있다"
- 쇼케이스 관점에서 EDA(Kafka) 자체가 서비스 분리를 자연스럽게 포함

## What/Why 재정의

### What: 이게 정확히 무엇인가?

**모호한 초기 정의:**
> "EDA와 MSA를 동시에 쇼케이스"

**해체 결과 - 3개의 별개 관심사:**

| 관심사 | 내용 | 2주 내 가능? |
|--------|------|-------------|
| **EDA** | Kafka 이벤트 기반 비동기 통신 | ✅ 가능 |
| **MSA 아키텍처** | 서비스 분리, 독립 실행 | EDA하면 자연스럽게 포함 |
| **Spring Cloud 인프라** | Gateway, Eureka, Config 등 | ❌ 별도 프로젝트 |

**최종 What 정의:**
> ✅ **"기존 동시성 제어 PoC에서 트래픽 증가로 후속 처리 병목 발생 → EDA 도입으로 해결"**

**EDA PoC가 증명하는 것:**
- ✅ Kafka 사용 경험 (JD 키워드 직접 충족)
- ✅ 이벤트 기반 비동기 처리
- ✅ 서비스 분리 (MSA의 기본, 자연스럽게 포함)
- ✅ 시스템 간 느슨한 결합
- ✅ **문제 기반 기술 도입 판단력** (Why → What → How 순서)

**EDA PoC가 증명하지 않는 것 (→ 모니터링/로깅 PoC로 분리 검토):**
- ❌ 모니터링 인프라 (Prometheus + Grafana)
- ❌ 낙관적 락 모니터링 (Retry 추적, 충돌률)
- ❌ 로깅 규칙 설계 (구조화된 로깅, 분산 추적)
- ❌ Spring Cloud 인프라 (Gateway, Eureka 등) → 우선순위 하락
- ❌ 모듈러 모놀리식 전환 → 로드맵으로만 언급 (범위 통제)

### Why: 왜 해야 하는가?

**이직 시장 요구:**
- JD 키워드: "MSA 경험 우대", "Kafka 사용 경험"
- EDA PoC만으로도 "Kafka 사용 경험" 충족
- Spring Cloud 인프라 없이도 충분히 어필 가능

**포트폴리오 시리즈 전략:**
1. ✅ 동시성 제어 PoC (완료) - "대규모 트래픽 처리"
2. 🔜 EDA PoC (이번 프로젝트) - "Kafka 이벤트 기반 비동기 처리"
3. 📋 모니터링/로깅 PoC (다음 프로젝트) - EDA 완료 후 2W 브레인스토밍으로 범위 확정

**로드맵 변경 이유 (MSA → 모니터링/로깅):**
- Spring Cloud MSA 쇼케이스는 설정 위주라 "경험"이라 하기 애매
- 동시성 PoC에서 자연스러운 확장: 낙관적 락 모니터링 (Retry 횟수, 충돌률 추적)
- 모니터링/로깅은 실무에서 매일 쓰는 실전 역량
- JD에서도 관측 가능성(Observability) 키워드 강세
- 포트폴리오 스토리: "문제 생성(동시성) → 비동기 확장(EDA) → 운영 가능하게(모니터링)"
- ⚠️ 단, 구체적 범위는 EDA 완료 후 `/2w-brainstorm`으로 재정의 (가설 기반 확정 방지)

**최종 산출물: 포트폴리오 (블로그가 아닌)**
- 블로그: "이걸 배웠다" → 학습 기록
- **포트폴리오: "이 사람에게 이 일을 맡길 수 있겠다" → 능력 검증**
- GitHub: 재현 가능한 코드 + 포트폴리오 시리즈 (동시성 → EDA → 모니터링)
- 면접: "이런 문제가 있어서 이런 판단을 했고, 이런 트레이드오프를 감수했습니다" (데이터 기반)

|------|------|
| **시간** | 2주 |
| **협업** | 혼자 |

| **최종 산출물** | **포트폴리오** (채용 시장에서의 능력 검증) |

**최소 구현 구조:**

Kafka를 쓴다고 Spring Cloud MSA가 필요한 건 아님.
장애 격리/독립 스케일링 시연을 위해 Producer/Consumer가 **별도 프로세스**이면 충분.

```
[PoC 최소 구조: 4개 서브 프로젝트 + Docker Compose]

동기 페어링 (Before: REST 직접 호출)
┌──────────────────────┐  REST   ┌─────────────────────────┐
│  order-service-sync  │ ──────→ │  inventory-service-sync  │
│  (Spring Boot)       │  직접   │  (Spring Boot)           │
│  Port: 8080          │  호출   │  Port: 8081              │
└──────────────────────┘        └─────────────────────────┘

비동기 페어링 (After: Kafka 이벤트)
┌──────────────────────┐  Kafka  ┌─────────────────────────┐
│  order-service-async │ ──────→ │  inventory-service-async │
│  (Spring Boot)       │  이벤트 │  (Spring Boot)           │
│  Port: 8082          │  발행   │  Port: 8083              │
└──────────────────────┘        └─────────────────────────┘

공통 인프라 (Docker Compose):
- kafka (+ zookeeper)
```

**왜 4개 서브 프로젝트인가:**
- 브랜치 분기보다 **나란히 놓고 즉시 비교** 가능
- 동일 시나리오를 동기/비동기 각각 실행 → TPS, Latency 정량 비교
- 포트폴리오에서 "같은 문제를 두 가지 방식으로 풀었다"가 시각적으로 명확

**왜 Spring Cloud가 필요 없는가:**

| 필요한 것 | Spring Cloud 없이 가능? | 설명 |
|----------|----------------------|------|
| 서비스 2개 별도 프로세스 실행 | ✅ | Spring Boot 2개 + Docker Compose |
| Kafka 이벤트 통신 | ✅ | spring-kafka 의존성만 추가 |
| 장애 격리 시연 | ✅ | 한쪽 컨테이너 중단으로 시연 |
| 서비스 디스커버리/라우팅 | 불필요 | 서비스 2~3개에서는 오버엔지니어링 |

**완성의 정의:**
- ✅ 정량 지표 측정 (처리 시간, 메시지 처리량, 동기 vs 비동기 비교)
- ✅ 재현 가능 (Docker Compose로 Kafka + 서비스 2개 즉시 실행)
- ✅ 포트폴리오 문서화 (Why/What/How + 트레이드오프 판단 근거)
- ✅ 명확한 결론 ("동기 vs 비동기, 어떤 상황에 어떤 방법")

## 가상 시나리오 (Virtual Scenario)

### 왜 가상 시나리오인가?

포트폴리오에서 "EDA를 학습했습니다"가 아니라 **"이런 문제가 있어서 EDA로 해결했습니다"**를 보여주기 위함.
면접관이 보는 차이: 기술을 "쓸 줄 아는" 사람 vs 기술을 **"왜 쓰는지 아는"** 사람.

### 시나리오: 이커머스 주문 프로세스의 팬아웃 문제

**Before State (문제 상황):**
> 이커머스 플랫폼에서 주문이 성공하면 알림, 결제, 배송, 포인트 등 여러 후속 처리가 필요하다.
> 동기 방식에서는 이 후속 처리들이 **직렬로 묶여 있어**,
> 트래픽 증가 시 응답 지연 및 장애 전파가 발생한다.

```
[Before: 동기 방식 - 모든 후속 처리가 직렬]

사용자 → 주문API → 재고차감 → 알림 → 결제 → 배송요청 → 포인트 → 응답
                                                            │
                                            여기까지 기다려야 응답

문제:
- 알림 서비스가 느리면 → 전체 주문이 느려짐
- 포인트 서비스가 죽으면 → 주문 자체가 실패 (Cascading Failure)
- 트래픽 3배 → 모든 후속 서비스가 동시에 3배 부하
```

**After State (EDA 도입 후):**
```
[After: 비동기 방식 - 핵심 트랜잭션과 후속 처리 분리]

사용자 → 주문API → 재고차감 → 응답 (빠름)
                      │
                      └─ OrderCompleted 이벤트 발행 (Kafka)
                              │
                    ┌─────────┼─────────┐
                    ▼         ▼         ▼
                  알림      결제      배송요청
                (독립)    (독립)     (독립)

효과:
- 알림이 느려도 → 주문 응답에 영향 없음
- 포인트 서비스가 죽어도 → 주문은 성공, 복구 후 재처리
- 트래픽 3배 → 각 Consumer가 독립적으로 스케일링
```

**핵심 도입 명분:**
- **1순위: 대규모 트래픽 증가 대응** — 사례 조사에서 배민/LINE/올리브영 모두 트래픽이 촉발점
- **2순위: 강결합 해소 (장애 전파 방지)** — 쿠팡의 Cascading Failure 문제
- 이 시나리오가 사례 5개 중 4개(배민, 쿠팡, 올리브영, LINE)와 동일한 패턴

**동시성 제어 PoC와의 관계:**
> 재고 차감(Redis Lua)은 동기든 비동기든 그 자체로 빠르다. 재고 차감에 EDA가 필요한 시나리오는 명분이 부족하다.
> EDA가 해결하는 문제는 **"주문 성공 후 여러 서비스에 알려야 하는 팬아웃(fan-out) 문제"**이며,
> 이는 동시성 제어와는 **별개의 문제**다.
>
> 따라서 별도 프로젝트로 분리하되, 포트폴리오 시리즈로 연결:
> "1. 재고 정합성(동시성) → 2. 주문 프로세스 확장성(EDA) → 3. 운영 관측(모니터링)"
> 기술적 필연성이 아닌, **같은 이커머스 도메인에서 단계별로 다른 문제를 풀었다**는 스토리.

**사례와의 대응:**

| 가상 시나리오 | 실제 사례 |
|--------------|----------|
| 후속 처리 동기 직렬 → 응답 지연 | 배민: 쓰기 처리량 한계 도달 |
| 한 서비스 장애 → 전체 실패 | 쿠팡: Cascading Failure |
| 트래픽 폭증 시 전체 서비스 품질 저하 | 올리브영: 올영세일 기간 서비스 저하 |

### 프로젝트 분리: 동시성 제어 vs EDA

같은 이커머스 도메인이지만 **해결하는 문제가 다르므로 별도 프로젝트**:

| 프로젝트 | 해결하는 문제 | 기술 | 상태 |
|----------|-------------|------|------|
| **동시성 제어 PoC** | 재고 차감의 정합성 | Redis Lua | ✅ 완료 |
| **EDA PoC** | 주문 후 후속 처리의 팬아웃/장애 격리 | Kafka | 🔜 이번 프로젝트 |

### 범위 통제: 모듈러 모놀리식은 로드맵으로

모놀리식 → 모듈러 모놀리식 전환은 별도 프로젝트 규모.
이번 PoC에서는 EDA 핵심 가치 검증에 집중하고, 모듈러 모놀리식은 로드맵으로만 제시한다.

> **"전부 다 하겠다"보다 "범위를 통제할 줄 안다"가 시니어 역량.**

**포트폴리오 시리즈 로드맵 (각각 독립 프로젝트, 같은 이커머스 도메인):**
1. ✅ 동시성 제어 PoC (완료) — 재고 차감의 정합성
2. 🔜 EDA PoC (이번) — 주문 후 후속 처리의 팬아웃/장애 격리
3. 📋 모니터링/로깅 PoC (별도)
4. 📋 모듈러 모놀리식 전환 (로드맵으로 제시)

## EDA 개념 학습 범위

### 이번 PoC에서 구현하는 개념

| 개념 | 설명 | 구현 이유 |
|------|------|----------|
| **Producer/Consumer** | 이벤트 발행/구독 기본 | EDA의 기본 단위 |
| **Topic/Partition** | Kafka 메시지 라우팅 | 순서 보장 + 병렬 처리 |
| **Consumer Group** | 다중 Consumer 부하 분산 | 스케일링 시연 |
| **At Least Once + 멱등성** | 메시지 최소 1회 전달 + 중복 방지 | 배민이 Exactly Once를 오히려 제거한 교훈 |
| **Dead Letter Queue (DLQ)** | 실패 메시지 격리 | 장애 격리 시연 (쿠팡 패턴) |
| **Partition Key** | 같은 키의 이벤트 순서 보장 | 배민: 주문별 이벤트 순서 유지 |

### 구현하지 않지만 알아야 할 개념

| 개념 | 설명 | 안 하는 이유 | 면접 대비 포인트 |
|------|------|-------------|-----------------|
| **Exactly Once Semantics** | 메시지 정확히 1회 전달 | 배민이 안정성 위해 **제거**함, 복잡도 대비 가치 낮음 | "왜 안 썼는가" 설명 가능해야 함 |
| **Transactional Outbox** | DB 트랜잭션 + 이벤트 발행 원자성 | Debezium 학습 비용 높음 (배민도 전담팀 필요) | 패턴의 목적과 작동 원리 이해 |
| **Event Sourcing** | 상태 대신 이벤트 이력 저장 | 올리브영도 Streams 조인 간소화함, PoC 범위 초과 | CQRS와의 관계, 적용 시나리오 |
| **CQRS** | 명령/조회 모델 분리 | PoC 범위 초과 | Event Sourcing과 함께 설명 가능 |
| **Saga Pattern** | 분산 트랜잭션 보상 처리 | 2주 제약 내 구현 어려움 | Choreography vs Orchestration 차이 |
| **Schema Registry** | 이벤트 스키마 버전 관리 | 초기 PoC에서는 JSON 직렬화로 충분 | 스키마 진화 문제 인지 |
| **Kafka Streams** | 스트림 처리 라이브러리 | LINE/올리브영 사례지만 이번 PoC 범위 밖 | 라이브러리 vs 프레임워크 차이 |
| **Kafka Connect / Debezium** | CDC 기반 이벤트 발행 | 학습 비용 높음, 애플리케이션 레벨로 대체 | CDC 패턴의 장점 이해 |

**상세 리서치:** [`eda-kafka-case-studies.md`](./eda-kafka-case-studies.md)

### 수집된 사례

### 공통 도입 패턴 (도입 인사이트)

**"언제 도입하는가":**
- **모놀리스 → MSA 전환** 시 강결합 해소가 핵심 동기 (쿠팡, 올리브영)
- **DB 부하가 병목**일 때 이벤트 기반 실시간 처리로 전환 (올리브영: 86% DB 호출 감소)
- **대규모 트래픽 증가**에 대응할 때 (배민: 300만 건/일, LINE: 100만 msg/sec)
- **점진적 도입이 원칙** - Big Bang 전환한 기업은 없음. 핵심 도메인 1~2개부터 시작

**기술 선택 패턴:**
- Kafka가 사실상 표준 (배민, LINE, 올리브영, 카카오, 네이버파이낸셜)
- 쿠팡만 유일하게 인하우스 MQ (특수한 트랜잭션 분해 요구)
- **Kafka Streams** = 라이브러리 방식이라 기존 서비스에 통합 용이 (LINE이 Samza에서 전환한 이유)

### 공통 트레이드오프

| 트레이드오프 | 사례 근거 | 심각도 |
|-------------|-----------|--------|
| **디버깅 복잡도 급증** | 배민: 2년간 41건 장애, 글로벌: 디버깅 시간 3배 | 🔴 높음 |
| **Exactly Once는 오히려 독** | 배민이 안정성 위해 **제거**함 | 🔴 높음 |
| **Kafka 설정 무지 = 장애** | 사람인: `max.poll.interval.ms` 몰라서 중복 처리 | 🟡 중간 |
| **전담 인력/팀 필수** | LINE: 전담 Kafka 팀, 배민: 전담팀 지원 | 🟡 중간 |
| **운영 비용 40% 증가** (부적절한 도입 시) | 글로벌 사례: $300K 투자 → 디버깅 3배 | 🔴 높음 |

### 평가 인사이트 (→ PoC 측정 지표에 직접 반영)

**사례에서 실제로 측정한 지표들:**

| 기업 | 측정 지표 | 실제 수치 |
|------|----------|-----------|
| 배민 | 일간 주문 건수, 트래픽 처리 배수 | 300만/일, 1x→3x |
| 배민 | 장애 건수 | 2년간 41건 (치명적 3건) |
| 쿠팡 | 일간 배포 횟수, 장애 격리 범위 | 100~200 서비스/일 |
| LINE | TPS (초당 메시지) | 100만 msg/sec |
| 올리브영 | **DB 호출 감소율** (Before/After) | **86% 감소** |

**PoC에서 측정해야 할 핵심 지표 (사례 기반 도출):**

| KPI | 왜 이 지표인가 | 측정 방법 |
|-----|---------------|----------|
| **TPS (처리량)** | 모든 사례의 기본 지표 | k6 부하 테스트, 동기 vs 비동기 비교 |
| **Latency (p50/p95)** | 실제 사용자 체감 성능 | End-to-end 이벤트 전달 시간 |
| **장애 격리** | 쿠팡/배민의 핵심 도입 동기 | Consumer 중단 시 Producer 100% 정상 여부 |
| **Consumer Lag** | Kafka 운영 건강도의 핵심 | Kafka 모니터링 |
| **메시지 유실율** | 신뢰성 증명 | 발행 건수 vs 소비 건수 비교, 0% 목표 |

**동시성 제어 PoC에서 배운 부하 테스트 교훈:**

> 이전 PoC에서 실제 환경과 비슷하게 커넥션 풀을 50으로 설정했지만,
> k6 동시 요청은 10만 개 수준이 무리라 훨씬 적은 동시성으로 테스트.
> 동시 요청 대비 커넥션 풀이 너무 여유로우니 비관락/낙관락/Redis/Lua **전부 널널**해져서
> 테스트 결과가 의도대로 나오지 않음.
> **커넥션 풀을 10개로 줄이니 차이가 명확히 보임.**

**적용 원칙: 자원 제한을 통한 명확한 차이 확보**
- PoC 환경에서는 실제 환경 모방보다 **자원을 의도적으로 제한**해서 차이를 극대화하는 것이 핵심
- 자원이 널널하면 동기/비동기 차이가 안 보임 → 어디서든 빠르게 처리되니까
- **병목을 만들어야 아키텍처 차이가 드러남**

| 자원 | 제한 방향 | 이유 |
|------|----------|------|
| **후속 서비스 응답 시간** | 500ms~2s 지연 | 동기에서는 직렬 대기, 비동기에서는 영향 없음 → 차이 극대화 |
| **동시 요청 수 vs 처리 자원** | 동시 요청 > 처리 용량이 되도록 설정 | 여유로우면 둘 다 빨라서 차이 안 보임 |
| **Consumer 처리 속도** | 의도적으로 느린 Consumer | Consumer Lag 발생 → 모니터링 지표 시연 가능 |

**이번 시나리오의 장점: 지연이 현실적으로 자연스러움**
> 동시성 PoC에서는 재고 차감이라는 단순 로직에 100ms sleep을 걸기 위해
> "비즈니스 로직이 있었다"는 인위적인 시나리오가 필요했다.
>
> 이번 EDA PoC에서는 그런 인위적 설정이 필요 없다:
> - 알림 서비스 → 외부 SMTP 호출 (수백ms 자연스러움)
> - 결제 서비스 → PG사 API 호출 (수백ms~수초 자연스러움)
> - 배송 서비스 → 물류 시스템 연동 (수백ms 자연스러움)
>
> **후속 서비스의 지연은 현실에서도 당연한 것이므로, 테스트 시나리오의 신뢰도가 높다.**

### `/1h-agile-phase`에 전달할 인사이트

**구현 구조:**
- 4개 서브 프로젝트 (동기 페어링 2개 + 비동기 페어링 2개)
- Spring Boot 멀티 프로젝트 + Kafka + Docker Compose (Spring Cloud 불필요)
- 동일 시나리오를 동기/비동기로 나란히 비교

**다이어그램 반영:**
- 주문 → 후속 처리(알림, 결제, 배송) 팬아웃 흐름
- 동기(REST 직렬) vs 비동기(Kafka 이벤트) 두 구조 병렬 구성
- DLQ(Dead Letter Queue)를 장애 격리 시연에 활용

**평가 지표 반영:**
- TPS, Latency(p50/p95), Consumer Lag, 메시지 유실율 → 정량 지표
- 장애 격리 시연 → 정성 지표 (Consumer 다운 시 Producer 정상 동작)
- **동기 vs 비동기 수치 비교** 패턴 (올리브영 Before/After 방식)
- 자원 제한을 통한 명확한 차이 확보 (동시성 PoC 교훈 적용)
- 후속 서비스 지연은 현실적으로 자연스러움 (인위적 sleep 불필요)

**범위 반영 (In/Out/Deferred):**
- ✅ **In**: At Least Once + 멱등성 (배민이 Exactly Once를 제거한 교훈)
- ✅ **In**: 장애 격리 테스트 (쿠팡/배민의 핵심 도입 동기)
- ✅ **In**: 동기 vs 비동기 정량 비교 (4개 서브 프로젝트로 나란히 비교)
- ❌ **Out**: Event Sourcing, CQRS, Saga Pattern (올리브영도 Streams 조인을 간소화)
- ❌ **Out**: Kafka Connect/Debezium (배민도 전담팀 비용 언급, PoC에서는 애플리케이션 레벨 이벤트 발행)
- ❌ **Out**: 스키마 레지스트리 (초기 PoC에서는 JSON 직렬화로 충분)
- ❌ **Out**: Spring Cloud (Gateway, Eureka 등 — 서비스 2~3개에서는 오버엔지니어링)
- ❌ **Out**: 모듈러 모놀리식 전환 (별도 프로젝트 규모, 로드맵으로만 제시)

3. ✅ "EDA 없는 MSA는 빈 껍데기" 가설을 **검증하고 재정의**
4. ✅ 2주 제약에 맞춰 **EDA만 집중**하는 결정
5. ✅ 사례 조사 기반으로 **현실감 있는 가상 시나리오** 설정 (팬아웃 문제)
6. ✅ "구현할 개념 / 구현하지 않지만 알아야 할 개념" 명확 구분
7. ✅ 동시성 PoC와의 부자연스러운 연결을 **스스로 걸러냄** (Lua 스크립트 성능 이점 → 관련 없음 인지)

### 배운 점
1. ✅ EDA, MSA 아키텍처, Spring Cloud 인프라는 **별개의 관심사**
2. ✅ EDA를 하면 서비스 분리(MSA 기본)는 **자연스럽게 포함**됨
3. ✅ "둘 다 하고 싶다" → **"순서를 정해서 하나씩"**이 이전 교훈의 반복 적용
4. ✅ MSA의 교과서적 목적은 "시장 대응 속도"이지만, **실제 도입 촉발점은 "트래픽/안정성 한계"**
5. ✅ MSA → EDA는 필연이 아니라 **"자주 발생하는 진화 경로"**
6. ✅ 포트폴리오에서 중요한 건 "기술 사용"이 아니라 **"문제 기반 기술 도입 판단"**
7. ✅ 모듈러 모놀리식 전환은 별도 프로젝트 규모 → **로드맵으로 범위 통제**가 시니어 역량
8. ✅ 재고 차감에 EDA를 연결하는 것은 **명분 부족** → 해결하는 문제가 다르면 별도 프로젝트
9. ✅ PoC 부하 테스트에서 **자원이 널널하면 아키텍처 차이가 안 보임** → 의도적 자원 제한 필수
10. ✅ 동시성 PoC에서는 sleep이 인위적이었지만, EDA에서는 **후속 서비스 지연이 현실적으로 자연스러움**
11. ✅ 오픈소스 기증은 이타적 행위가 아닌 **전략적 투자** (유지보수 분산 + 인재풀 + 표준화)

### 핵심 교훈
> **"EDA와 MSA를 동시에"가 아니라, "EDA(2주) → MSA(2주)"로 분리하면 각각 완성 가능하다.**
> **이전 동시성 PoC에서 배운 "하나를 깊게" 원칙이 그대로 적용된다.**
> **"이걸 배웠다"(블로그)가 아니라 "이런 문제를 이렇게 판단했다"(포트폴리오)가 채용 시장에서 강력하다.**
> **기술 간 연결에 명분이 부족하면 무리하게 묶지 말고 별도 프로젝트로 분리하라.**
